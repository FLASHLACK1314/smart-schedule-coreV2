## 自动排课 API

### 接口信息

```
POST /api/auto-schedule/execute
Content-Type: application/json
```

### 请求参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| semesterUuid | String | ✅ | 学期 UUID |
| courseClassMapping | Map<String, List<String>> | ✅ | 课程UUID → 行政班UUID列表 的映射 |
| buildingUuids | List<String> | ❌ | 限制使用的教学楼 UUID 列表 |
| teacherAssignment | Map<String, String> | ❌ | 课程UUID → 教师UUID 的指定 |
| teacherSelectionStrategy | String | ❌ | 教师选择策略：balanced(默认)/random/first |
| overwrite | Boolean | ❌ | 是否覆盖已有排课，默认 false |
| scheduleMode | Integer | ❌ | 排课模式：0=预览，1=正式 |
| populationSize | Integer | ❌ | 遗传算法种群大小，默认 100 |
| maxGenerations | Integer | ❌ | 最大迭代代数，默认 500 |
| crossoverRate | Double | ❌ | 交叉概率，默认 0.8 |
| mutationRate | Double | ❌ | 变异概率，默认 0.2 |
| eliteSize | Integer | ❌ | 精英保留数量，默认 10 |

### 请求示例

#### 基础示例：为2个班级排3门课

```json
{
  "semesterUuid": "学期UUID（从sc_semester表获取）",
  "courseClassMapping": {
    "数据结构课程UUID": ["计科2101班UUID", "计科2102班UUID"],
    "操作系统课程UUID": ["计科2101班UUID", "计科2102班UUID"],
    "大学体育课程UUID": ["计科2101班UUID"]
  },
  "overwrite": true,
  "scheduleMode": 1
}
```

#### 完整示例：指定教学楼和算法参数

```json
{
  "semesterUuid": "2024-2025学年第一学期UUID",
  "courseClassMapping": {
    "CS101课程UUID": ["计科2101班UUID", "计科2102班UUID"],
    "PE101课程UUID": ["计科2101班UUID", "计科2102班UUID"],
    "CS201课程UUID": ["软件2101班UUID"]
  },
  "buildingUuids": [
    "A栋教学楼UUID",
    "B栋教学楼UUID",
    "D栋教学楼UUID"
  ],
  "teacherAssignment": {
    "CS101课程UUID": "张教授UUID"
  },
  "teacherSelectionStrategy": "balanced",
  "overwrite": true,
  "scheduleMode": 1,
  "populationSize": 100,
  "maxGenerations": 300,
  "crossoverRate": 0.8,
  "mutationRate": 0.2,
  "eliteSize": 10
}
```

## 排课流程说明

### 1. 数据准备阶段

由于 Demo 模式没有预置教学班，排课时系统会：

1. **自动创建教学班**：根据 `courseClassMapping` 参数
2. **自动选择教师**：根据 `teacherSelectionStrategy` 策略
3. **验证教师资格**：确保教师有对应课程的授课资格

### 2. 算法执行阶段

遗传算法执行流程：

```
初始化种群 → 评估适应度 → 选择 → 交叉 → 变异 → 精英保留 → 迭代 → 输出最优解
```

### 3. 结果保存阶段

根据 `scheduleMode` 参数：
- `0` = 预览模式（status=0，可被下次排课覆盖）
- `1` = 正式模式（status=1，作为正式课表）

## 硬约束和软约束

### 硬约束（必须满足，否则惩罚 1,000,000 分）

| 约束类型 | 说明 |
|---------|------|
| 教师时间冲突 | 同一教师同一时间不能有多门课 |
| 教室时间冲突 | 同一教室同一时间不能有多门课 |
| 班级时间冲突 | 同一行政班同一时间不能有多门课 |
| 教室容量不足 | 教室容量必须大于等于学生人数 |
| 教室类型不匹配 | 课程类型必须与教室类型匹配 |
| 教师资格不符 | 教师必须有对应课程的授课资格 |
| 已有排课冲突 | 不能与已有正式排课冲突 |

### 软约束（优化目标，满足有奖励）

| 约束类型 | 奖励分数 |
|---------|---------|
| 教师时间偏好 | +1000 |
| 教师工作量均衡 | +500 |
| 课程分布均匀 | +300 |

## 常见问题和解决方案

### 问题1：排课结果有大量硬约束冲突

**可能原因**：
1. 课程类型-教室类型映射缺失
2. 教室资源不足
3. 教师资格未配置

**解决方案**：
- 检查 `sc_course_classroom_type` 表是否有对应映射
- 检查 `sc_course_qualification` 表是否有教师资格记录
- 减少排课数量或增加教室资源

### 问题2：部分教学班未被排课

**可能原因**：
1. 找不到合适的教室（容量或类型不匹配）
2. 时间槽资源耗尽

**解决方案**：
- 增加可用教学楼（`buildingUuids`）
- 减少行政班数量（降低合班人数）

### 问题3：排课速度慢

**解决方案**：
- 减少种群大小（`populationSize`）
- 减少最大迭代代数（`maxGenerations`）
- 分批次排课
